What it is

01

The art of making an AI agent
actually finish.

Goal Forge is a Codex skill — invoked via $goal-forge — that converts the vague spark of a coding idea into a rigorously structured GOAL.md contract. Not a plan. Not a to-do list. A contract.

The skill exists because long autonomous AI runs collapse under ambiguity. It enforces a sequence: interrogate, tighten, compile, verify. Nothing ships until every exit criterion is testable and every boundary is explicit.

Think of it as the editor who won't let you go to press until every word earns its place — relentless, structural, and completely necessary.

Core Doctrine

"Every done_when item must name a command, a file artifact, or an observable behaviour — so the agent knows, without asking, when to stop."

The Four
Stages

Each stage is a discipline. Each gate is non-negotiable. The sequence cannot be shortcut.

01

Stage One / Discovery

Interview Mode

Open-ended, Socratic questioning designed not to confirm what you think you want — but to expose what you haven't decided yet. The AI reads an existing SPEC.md if present, or treats the raw idea as seed material, then asks the questions that reveal scope, architecture, and edge cases.

Critical Gate

Will not advance to compile until done_when contains user-approved, measurable completion criteria. No approximations accepted.

  • Forces decisions on scope boundaries
  • Avoids "obvious" checklist questions
  • Asks only what exposes genuine ambiguity
  • Loops until criteria are unambiguous
02

Stage Two / Critique

Tighten Mode

The spec that survived Interview is not finished — it is merely a candidate. Tighten Mode attacks it skeptically: surfacing every ambiguity, presenting dual interpretations side by side, and recommending the most maintainable resolution. Nothing is updated until choices are unambiguous.

Output Requirements

Final spec must include: goals, non-goals, architecture, edge cases, risks, verification commands, and done_when. All eight. No exceptions.

  • Presents two distinct interpretations per ambiguity
  • Explains why each ambiguity matters
  • Routes ambiguous specs back to Interview
  • Recommends the most maintainable approach
03

Stage Three / Compilation

Compile Mode

The tightened SPEC.md is transformed into GOAL.md using an eight-block XML structure. This is the document that Codex actually executes. Every block maps to an agent behaviour: what to read, what to build, how to verify, when to stop. Weak specs are rejected and routed backward.

Rejection Criteria

Specs lacking measurable done_when, boundaries, verification commands, or repo context are refused and returned to earlier stages with specific deficiencies listed.

  • Maps spec into eight XML goal blocks
  • Self-checks measurability before output
  • Saves only the block body — no wrapper markup
  • Verifies execution-orientation of the goal
04

Stage Four / Readiness

Check Config Mode

Before a multi-hour autonomous Codex run begins, the environment itself must be verified. The skill runs scripts/inspect_codex_config.py — a read-only reporter that validates model version, context window, reasoning effort, approval policy, and sandbox access. One status: ready or not ready.

Validated Settings

Reports version, config keys, trusted-project status, and all gaps against the required autonomous configuration profile.

  • Checks gpt-5.5 model with 1,050,000-token window
  • Validates reasoning effort levels
  • Confirms approval policy and sandbox mode
  • Reports autonomous_goal_status: ready

The Eight
Blocks

Every GOAL.md is structured into eight XML blocks. Together they form a complete contract — the deliverable, the context, the constraints, and the precise conditions of completion.

01
<goal>
Goal

Deliverable description. Specific, measurable, execution-focused. Describes what to build — not how to build it. No vague terms without metrics.

02
<context>
Context

Required reads, system inspections, discovery commands. Concrete file paths where known; rg / find commands when uncertain.

03
<constraints>
Constraints

Architecture rules, non-goals, risk boundaries, and anti-patterns. What the agent must never do is as important as what it must do.

04
<done_when>
Done When

Concrete completion criteria. Each item names a command, file artifact, or observable behaviour. The agent determines completion autonomously against these.

05
<workflow>
Workflow

Step-by-step execution order with parallelisation notes. Structured phases: inspect → plan → implement → verify → refine → review.

06
<verification_loop>
Verification Loop

Post-change commands and fallback checks. Leads with focused checks, progresses to broad validation. Specifies manual QA evidence thresholds.

07
<execution_rules>
Execution Rules

Repo hygiene, editing standards, testing requirements. The standing operating procedures that apply to every action in the run.

08
<output_contract>
Output Contract

Final artifacts and response format. What the agent must produce and how it must present the result. The shape of done.

Autonomous
Configuration
Profile

For multi-hour sessions, Goal Forge targets a specific Codex environment. These are not suggestions — they are prerequisites for the kind of extended, unsupervised execution that the /goal contract is designed to enable.

The model must be gpt-5.5 with a 1,050,000-token context window. Reasoning effort must be high during execution, xhigh during planning. Auto-compaction triggers at 997,500 tokens.

Safety note: approval_policy = "never" and sandbox_mode = "danger-full-access" must only be applied to explicitly trusted projects. These settings grant the agent full write access to the filesystem — they are not defaults. They are a conscious choice, made with full awareness of what they permit.

codex.toml — Target Autonomous Configuration

model                          = "gpt-5.5"
model_context_window           = 1_050_000
model_auto_compact_token_limit = 997_500

model_reasoning_effort         = "high"    # execution
plan_mode_reasoning_effort     = "xhigh"   # planning

approval_policy                = "never"
sandbox_mode                   = "danger-full-access"

[features]
goals                          = true
1,050,000 tokens Context window — enough for very long multi-file autonomous runs without losing early context
xhigh reasoning effort Applied during planning phases to maximise architectural foresight before writing code
approval_policy: never Trusted projects only — agent executes without interruption, relying entirely on the GOAL.md contract
